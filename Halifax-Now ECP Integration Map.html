Where the data
meets the design.

This document maps every field in The Events Calendar Pro and its Event Aggregator add-on to a specific element in the Halifax Now design. It's the contract between WordPress and the front-end — what lives where, what we need that ECP doesn't give us out of the box, and how to handle the tricky parts (moods, critic's picks, neighbourhoods).

The Aggregator

§ 02 · Import sources

Event Aggregator imports from external sources. Each source has a reliability profile — know which fields arrive clean and which need manual cleanup before publish.

  • 1
    iCal / Google Calendar Reliable: SUMMARY → title, DTSTART/DTEND, LOCATION → venue, DESCRIPTION → content.
    Missing: featured image, category, price. Queue for editor review before publish.
  • 2
    Eventbrite Reliable: title, dates, venue, price, image, URL (tickets). Maps well.
    Category mapping needs rules — Eventbrite's taxonomy is flatter than ours. Set a default per import source.
  • 3
    Meetup Reliable: title, dates, venue, group name (→ organizer). Usually no featured image.
    Group name is valuable — map to tribe_organizer so "North End Run Crew" shows on the card.
  • 4
    Facebook Events Scheduled deprecated by Meta, but still works via URL import. Treat as lossy — expect to hand-edit everything.
  • 5
    CSV / manual submission From the public Submit form (Community Events add-on). Moderate before publish. This is the primary path for small local events.

The moderation rule.

Every imported event lands in pending state. An editor approves it, adds the mood, neighbourhood, and (optionally) the critic flag. Nothing hits the front-end without a human touch. Halifax Now's voice depends on it.

What ECP doesn't give us.

§ 03 · Custom fields

These are the fields the design needs that aren't in ECP. Add them via Advanced Custom Fields (ACF) on the tribe_events post type. All optional except where noted.

hfx_moods (acf checkbox)

Moods

Multi-select from a fixed list. Drives the mood chip filter bar.

Values: chill, rowdy, date, kids, solo, crew, free, rainy
hfx_neighbourhood (acf select)

Neighbourhood

Single-select. Required if venue doesn't already have one attached. Drives the hood filter.

Values: North End, Downtown, South End, West End, Dartmouth, Bedford, Quinpool, Spring Garden
hfx_critic_pick (acf true/false)

Critic's Pick

Editorial flag. If true, card gets a ★ Pick stamp; event is eligible for the homepage Critic's Picks block.

~5–10% of weekly events. Editors set this during moderation.
hfx_short_blurb (acf text)

Short blurb

One-liner for cards. Max 90 chars. If blank, card falls back to post_excerpt truncated. Worth writing by hand for picks.

"Six comics, one hour, two-drink minimum."
hfx_editor_blurb (acf textarea)

Editor's blurb

The opinionated 2–3 sentence voice of the site. Shown in hero feature + detail page callout. Required on critic's picks.

"The Carleton kicks off ten days of jazz…"
hfx_hue (computed)

Image hue

Don't edit manually. Computed server-side from primary category. See §04 for the mapping.

music → 220 · comedy → 14 · arts → 280 …

The Flow

§ 05 · Request → render

Everything front-end hits the REST API — no PHP templates. Headless WordPress gives us the design system freedom. Here's the pipeline:

01 · Source
WordPress
ECP + ACF custom fields. Editor-moderated.
→
02 · API
REST /events
Normalizes ECP + ACF into a flat JSON shape the front-end expects.
→
03 · Render
Front-end
React component library applies treatments. Shown here in Halifax Now colours.

The event JSON contract.

The front-end expects this shape per event. Everything is nullable except id, title, date, venue, category.

{ id, title, short, blurb, date, time, endTime, venue, address, hood, category, mood[], image, hue, price, priceLabel, critic, recurring, url }

The Gaps

§ 06 · Known shortfalls

Things ECP doesn't do well that we'll need to handle elsewhere.

  • ⚠
    Event images are inconsistent by default. ECP doesn't enforce a minimum size or aspect ratio. Our treatment masks most of this, but tiny images (< 400×300) still pixelate. Fix: upload validation on Submit form + resizing filter on import.
  • ⚠
    Multiple-day events need special handling. A three-day festival is one post with one start + end. The front-end splits it visually (showing Fri, Sat, Sun cards) but deep-links to the same detail page. Check that card rendering handles this cleanly.
  • ⚠
    Recurring events don't generate post IDs per occurrence. "Every Sunday Run Club" is one post. Calendar heatmap and filter logic need to expand occurrences on the fly. Performance note: cache this.
  • ⚠
    Cancellations aren't a first-class concept. ECP has post_status: trash or unpublish — neither tells the user "this was cancelled." Fix: add hfx_status ACF with values active · cancelled · postponed. Cancelled events stay visible with a strike-through for 48 hours.
  • ⚠
    No built-in "editor's note." If the critic wants to update a pick mid-week ("new opening act confirmed"), ECP has no concept of this. Fix: hfx_update_note ACF, shown as a small yellow banner above the blurb on the detail page.

The Checklist

§ 07 · Dev handoff

Before front-end development starts.

1. ACF installed & custom fields registered on tribe_events (§03)

2. REST endpoint /wp-json/hfx/v1/events returning normalized JSON (§05)

3. Category slugs locked to the 12 in §04 (hue mapping is slug-based)

4. Moderation workflow live — imports land in pending, editors approve with moods + hood

5. Venue geocoding enabled (ECP Pro setting) — required for map

6. Upload min-size validation on Community Events submit form
